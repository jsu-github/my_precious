# Research Summary — v1.3: Crypto Asset Tracking

**Synthesized:** 2026-04-13

## Executive Summary

v1.3 is an **extension milestone, not a new foundation**. The existing stack handles everything — zero new npm packages are required. The core work is three DB migrations, one new API route, two extended backend handlers, and one modified frontend component (AssetModal) with a new spot price sub-panel. An entire class of downstream screens (Dashboard, Ledger, Tier, Analytics) update automatically once `recalculate` is extended, requiring no direct code changes.

The central architectural decision is a **valuation branch split** in `POST /api/valuation/recalculate`: tokenized gold (PAXG, XAUT) derives value from the XAU spot table established in v1.2, while native crypto (XMR, BTC) derives value from a new `crypto_spot_prices` table. This aligns with the dashboard's sovereign risk framing — tokenized gold should consolidate with physical gold, not float independently as a separate price feed.

The single highest-risk dependency is migration number collision with v1.2. Migration numbers must be assigned only after v1.2 is merged, not during planning. Every other risk is containable with the guards documented in the pitfalls research.

## 2. Feature Table Stakes vs. Differentiators

### Table Stakes (must ship — absence makes crypto tracking feel broken)

| Feature | What It Does | Complexity |
|---------|-------------|------------|
| `coin_symbol` + `coin_name` fields | Canonical ticker for price lookup; display name separate from asset `name` | Low — two VARCHAR columns via migration |
| `custody_type` field | Exchange / hot_wallet / hardware_wallet / paper_wallet / custodian | Low — one VARCHAR + dropdown in AssetModal |
| `crypto_spot_prices` table + UI | Manual EUR/coin entry; append-only; same UX as v1.2 metals spot prices | Low |
| Gold-backed token valuation (PAXG, XAUT) | Derive EUR value from `qty_oz × 31.1035 × xau_spot_per_gram` — NOT from token market price | Medium — conditional branch in recalculate |
| Native crypto valuation (XMR, BTC, ETH) | `qty × price_per_unit` from `crypto_spot_prices` | Low |
| Computed value propagation | Dashboard / Ledger / Tier / Analytics automatically reflect correct crypto values after recalculate | Medium — must coordinate with v1.2 effective-value pattern |

### Differentiators (add value, build after table stakes solid)

| Feature | Value Proposition | Priority | Complexity |
|---------|-------------------|----------|------------|
| Gold oz equivalent display | PAXG/XAUT rows show "2.0 oz gold equivalent" — reveals true gold exposure | High | Low (derived from qty) |
| Combined gold exposure summary | Analytics: physical oz + PAXG oz + XAUT oz = total gold oz | High | Low |
| Custody risk indicator badge | Visual badge differentiating self-custody vs. counterparty | Medium | Low (from existing `custody_type`) |
| Network / chain field | `network VARCHAR(30)`: 'Ethereum', 'Tron', 'Monero' | Low | Low |
| Contract address storage | Read-only ERC-20 reference for manual Etherscan verification | Low | Low |

### Defer to v1.4+

Network/chain field, contract address storage — relevant but not blocking the core tracking use case.

## 3. Key Architecture Decisions

### Migration Sequence

Migrations must be numbered AFTER v1.2 is merged. Planning uses placeholders.

| Placeholder | Purpose | Type |
|-------------|---------|------|
| `02X_acquisition_qty_precision.ts` | Widen `acquisitions.quantity` DECIMAL(20,6) → (20,12) | ALTER TABLE (raw SQL) |
| `02Y_crypto_asset_fields.ts` | ADD `coin_symbol`, `coin_name`, `custody_type` to `assets` | ALTER TABLE |
| `02Z_crypto_spot_prices.ts` | CREATE `crypto_spot_prices` table + index | CREATE TABLE |

**First action at implementation time:** `ls api/migrations/ | tail -5` — verify the highest existing number before creating any file.

### Valuation Branching (the core architectural rule)

```
POST /api/valuation/recalculate
  │
  ├─ Branch 1: asset_class = 'precious_metals'          (v1.2 — unchanged)
  │    value = qty × toGrams(weight) × spot_prices(sub_class)
  │
  ├─ Branch 2: asset_class = 'crypto' AND sub_class = 'gold'   ← PAXG, XAUT
  │    value = SUM(qty_oz) × 31.1035 × spot_prices WHERE metal = 'gold'
  │    Guard: no XAU entry → skip silently (preserve current_value)
  │
  ├─ Branch 3: asset_class = 'crypto' AND sub_class ≠ 'gold' AND coin_symbol NOT NULL
  │    value = SUM(qty) × crypto_spot_prices(symbol)    ← XMR, BTC
  │    Guard: no symbol entry → skip silently (NEVER zero out current_value)
  │
  └─ Branch 4: all other classes → skip (manual current_value unchanged)
```

**Branch discriminator:** `sub_class = 'gold'` on a crypto asset is the trigger for the XAU path. This is the single most important invariant to enforce and test.

**PAXG/XAUT fixed convention:**
- `acquisitions.quantity` = number of tokens (1 token = 1 troy oz — no conversion needed)
- `weight_per_unit = NULL`, `weight_unit = NULL` — crypto assets never use the metals weight fields
- PAXG/XAUT symbols must NEVER appear in `crypto_spot_prices`

### Build Order (Wave Dependency Graph)

```
Wave 1 — Database (no dependencies)
  acquisitions.quantity precision migration
  02Y_crypto_asset_fields.ts
  02Z_crypto_spot_prices.ts

Wave 2 — API (depends on Wave 1)
  cryptoSpotPrices.ts (new route)
  valuation.ts (extend recalculate — Branch 2 + 3)
  routes/index.ts (mount new router)

Wave 3 — TypeScript types + API client (depends on Wave 2 contracts)
  types.ts: CustodyType, CryptoSpotPrice, extend Asset
  api.ts: cryptoSpotPrices namespace

Wave 4 — Asset Management UI (depends on Wave 3)
  AssetModal.tsx: crypto-conditional fields + PAXG/XAUT hint

Wave 5 — Crypto Spot Price UI (depends on Wave 3)
  Crypto Prices panel inside v1.2 spot price section

Wave 6 — Verification
  Create XMR asset → spot price → recalculate → verify Dashboard + Ledger + Tier
  Create PAXG asset (sub_class='gold') → verify XAU path used, NOT crypto_spot_prices
  Existing PM assets: recalculate regression (Branch 1 untouched)
```

**Critical ordering constraint:** AssetModal must not ship before `types.ts` — `CustodyType` must exist in `Asset` before the modal compiles.

### What Stays Untouched

`assets.ts`, `dashboard.ts`, `ledger.ts`, `tierConfig.ts` routes require no changes.
`DashboardPage.tsx`, `LedgerPage.tsx`, `TierPage.tsx`, `AnalyticsPage.tsx` require no changes.
All existing migrations 001–021 must never be modified.

## 4. Top 5 Critical Pitfalls

### P1: Migration Number Collision with v1.2 (BLOCKING)

v1.2 is in progress and will consume migration numbers that v1.3 plans to use. If v1.3 creates migration `021_...ts` and v1.2 already ran `021_spot_prices.ts`, Knex silently skips the v1.3 migration — the schema is incomplete with no error.

**Prevention:** Assign v1.3 migration numbers only after v1.2 merges. Use placeholder names (`02X`, `02Y`, `02Z`) during planning. First action at implementation: `ls api/migrations/ | tail -5`.

---

### P2: PAXG/XAUT Routed to `crypto_spot_prices` Instead of XAU Path

Developer or user enters PAXG/XAUT in the crypto spot prices table. Valuation uses the token market price instead of XAU spot. The tokenized gold position no longer consolidates with physical gold — defeating the dashboard's primary differentiator.

**Prevention:** API validation rejects `PAXG`/`XAUT` symbols in `POST /api/crypto-spot-prices`. AssetModal shows "Value derived from XAU spot price" when `sub_class = 'gold'`. Branch 2 condition is tested explicitly in Wave 6 verification.

---

### P3: Cartesian Product from Un-CTEd `crypto_spot_prices` Join

`crypto_spot_prices` is append-only. A JOIN without isolating the latest-per-symbol row multiplies `SUM(qty × price)` by the number of historical price entries. Dashboard total doubles silently after each new spot price entry. The bug appears only after the second price row is inserted.

**Prevention:** Always use `DISTINCT ON (symbol) ORDER BY symbol, recorded_at DESC` in a CTE before joining. Regression test: insert 3 XMR prices, verify dashboard total doesn't change.

---

### P4: `acquisitions.quantity` Silent Truncation for Crypto

`DECIMAL(20,6)` truncates XMR entries with >6 decimal places silently — PostgreSQL rounds without error. This violates the codebase principle of precision for financial data.

**Prevention:** Widen to `DECIMAL(20,12)` via raw SQL `ALTER COLUMN TYPE` (Knex fluent API cannot change existing column precision). Apply as the first migration of v1.3, before any crypto acquisition is entered.

---

### P5: PAXG/XAUT Quantity Unit Mismatch (~31× valuation error)

If PAXG/XAUT quantities are stored in grams (copying the metals convention) instead of troy oz (token count), the formula `qty × 31.1035 × xau_spot` produces a result ~31× too large. The error magnitude can pass a casual eyeball check on a small position.

**Prevention:** Canonical rule: for PAXG/XAUT, `acquisitions.quantity` = number of tokens (= troy oz). Acquisition form hint: "1 token = 1 troy oz — enter number of tokens." `weight_per_unit` must be NULL on all crypto assets. The 31.1035 constant is hardcoded in the valuation formula, not derived from a field.

## 5. Milestone Scope Confidence

| Dimension | Assessment |
|-----------|-----------|
| **Overall complexity** | **Low-Medium** — no new patterns, no new libraries, no new pages |
| **DB schema changes** | 3 migrations: one ALTER (quantity precision), one ALTER (asset fields), one CREATE (crypto_spot_prices). All follow established patterns. |
| **API changes** | 1 new route file (mirrors existing spotPrices.ts). 1 extension to valuation.ts (adds 2 branches to existing 4-branch structure). 1 line in index.ts. |
| **Frontend changes** | 1 modal extension (AssetModal — conditional fields). 1 UI sub-panel (crypto spot prices). Zero new pages. Zero new navigation items. |
| **Passive propagation** | 4 screens (Dashboard, Ledger, Tier, Analytics) updated automatically — no code changes. Substantial value delivered at zero direct cost. |
| **Primary risk** | Migration number sequencing (dependency on v1.2 merge) — not technical complexity |
| **Secondary risk** | PAXG/XAUT valuation branch correctness — easy to get wrong, easy to test |

| **Confidence** | **HIGH** — all changes are additive extensions of patterns already present in the codebase. No speculative design decisions required. |
